TASK 3 — Recalibrate Tier 2 semantic detection across all 41 frameworks
Context
The current Tier 2 prompt (semanticFrameworkCheck in server/agents/frameworkDetector.ts:120-200) asks the LLM for a binary applied: true | false plus a confidence: high | medium | low. This two-axis design has two problems the diagnostic surfaced:

False negatives — the spot-check at Step 4(c) showed plausible implicit applications being rejected as applied: false for SWOT, Porter, and others. Students who restructured a decision around a framework's reasoning pattern without naming the framework were classified as not-applied.
No protection against false positives — if the LLM hallucinates (decides "this student must be applying SWOT because they mentioned a competitor"), there's no scale calibration that distinguishes "the student genuinely structured their reasoning around the framework" from "some keywords vaguely resemble the framework, so I'll say yes." The current confidence: low is the only output for "I'm not sure but I'll say yes" — and there's evidence the LLM is using it to default toward yes when in doubt.

The fix is a four-tier scale that forces the LLM to commit to a positive evidence threshold per detection AND surfaces "the student definitely did NOT apply this" as a distinct first-class outcome, not just "applied: false." The four tiers also let downstream calibration (FIX 1's confidence-weighted scoring) treat each tier appropriately.
This applies to ALL 41 frameworks in the registry, not just SWOT and Cost-Benefit. The current rubric's recognition signals are written at registry time and consumed by the same prompt for every framework. We're changing the prompt's decision scale, not individual framework rubrics — though we'll also tighten the registry-level recognition signals where they're vague.

3.2 Replace the binary scale with a four-tier evidence scale
Edit server/agents/frameworkDetector.ts semanticFrameworkCheck function (lines 120-200). Replace the applied: boolean + confidence: high|medium|low schema with a four-tier evidence_level enum:
"none"            — the student's reasoning shows NO structural alignment with this framework.
                    The framework's reasoning pattern is absent. Student's text may discuss the
                    same domain (e.g., a strategy decision) without using this specific framework.

"weak_implicit"   — the student's reasoning faintly resembles part of the framework's pattern,
                    but the alignment is incomplete or could be coincidental. One element of the
                    framework appears; the rest do not. A reasonable observer might or might not
                    call this an application.

"strong_implicit" — the student's reasoning structure clearly mirrors the framework's pattern
                    across multiple elements, even though the framework is not named. A reasonable
                    observer would say "this student is using <framework>'s lens" with confidence.
                    The reasoning is anchored in the framework's core concepts, just unnamed.

"explicit"        — the student names the framework directly OR uses canonical framework
                    vocabulary in a way that demonstrates correct application. Tier 1 (keyword)
                    typically catches these; Tier 2 only emits "explicit" when Tier 1 missed
                    a phrasing variation but the application is unambiguous.
The corresponding level mapping in the detector pipeline:

evidence_level: "none" → level: "not_evidenced" in the final detection
evidence_level: "weak_implicit" → level: "implicit", confidence: "low"
evidence_level: "strong_implicit" → level: "implicit", confidence: "medium"
evidence_level: "explicit" → level: "explicit", confidence: "high"

This is a structural change. Drop the old applied: bool and confidence fields from the LLM output schema. The downstream pipeline should map evidence_level to level + confidence at the boundary, not the LLM.
3.3 Rewrite the system prompt with positive-evidence-required wording and false-positive guards
Replace the EN system prompt (lines 162-170) with:
You determine whether a student's reasoning APPLIES specific analytical frameworks. You do not judge whether the student is correct or wrong — only whether the framework's reasoning pattern shows up in their text. You do not need the student to name the framework.

DECISION SCALE — pick exactly ONE for each framework:

- "none": the framework's reasoning pattern is absent. The student may discuss the same problem domain without using this framework's lens. Default to "none" if uncertain.

- "weak_implicit": one element of the framework's pattern appears, but the rest don't, or the alignment is partial enough that it could be coincidence. Use this when the student's reasoning hints at the framework but doesn't anchor on it.

- "strong_implicit": the student's reasoning is clearly structured around this framework's pattern, with multiple elements present, even though they don't name it. A professor reading the response would say "this student is thinking like <framework>." Reserve this for unambiguous structural mirroring.

- "explicit": the student names the framework directly, OR uses canonical framework vocabulary correctly. Most explicit cases are caught upstream (keyword tier); only emit "explicit" when you see a clear paraphrase of the framework's name or terminology that the keyword tier missed.

CRITICAL RULES TO PREVENT MISCALIBRATION:

1. POSITIVE EVIDENCE REQUIRED. To label "weak_implicit" or stronger, you MUST identify a specific phrase or sentence in the student's text that EXEMPLIFIES the framework's reasoning. Domain-shared vocabulary alone (e.g., "competitor" appearing in a Porter-eligible scenario) is NOT positive evidence — Porter's reasoning structure must show up.

2. NO BIAS TOWARD APPLICATION. Default to "none" when in doubt. False positives (saying a student applied a framework they didn't) are as harmful as false negatives. The professor uses this signal to grade — getting it wrong in either direction misrepresents the student.

3. MULTIPLE FRAMEWORKS, INDEPENDENT JUDGMENTS. The student's text may apply zero, one, or several frameworks. Judge each one independently. Do not assume that if one framework applies, others probably do too.

4. quotedReasoning MUST be a verbatim substring of the student's input — no paraphrase, no summary. Empty if evidence_level is "none".

5. explanation describes which conceptual element the student exercised (or, for "none", briefly says why the framework's pattern is absent). Observation only — never use evaluative language ("correct", "should have", "ideal", "missed an opportunity").
ES: equivalent translation, with the same five critical rules in Spanish. Keep evidence_level literal values in English (none | weak_implicit | strong_implicit | explicit) — the parser checks for those exact strings.

3.5 Update the parser
Find where the LLM verdict is consumed in frameworkDetector.ts (probably the loop that pushes detections after semanticFrameworkCheck returns). The current code reads applied and confidence. Update it to read evidence_level and map to level + confidence per the table in 3.2:
tsfunction mapEvidenceLevelToDetection(level: string): { level: "explicit" | "implicit" | "not_evidenced", confidence: "high" | "medium" | "low" | undefined } {
  switch (level) {
    case "explicit":         return { level: "explicit",      confidence: "high" };
    case "strong_implicit":  return { level: "implicit",      confidence: "medium" };
    case "weak_implicit":    return { level: "implicit",      confidence: "low" };
    case "none":
    default:                 return { level: "not_evidenced", confidence: undefined };
  }
}
Apply the §T-003B word-count floors AFTER this mapping (they currently downgrade medium → low on <15-word inputs and reject applied=true entirely on <10-word inputs). The floors still apply correctly to strong_implicit → medium on short inputs.
If a verdict has evidence_level: "none", do NOT push a not_evidenced detection from Tier 2 — fall through to Tier 3 signal-pattern as today. Keep Tier 3 fallthrough behavior (the existing test in calibration.semanticFloor.realpath.test.ts covers this).
If a verdict has evidence_level: "weak_implicit" AND the §T-003B word-count floor reduces confidence to "low" on a <15-word input, the result is still a recorded implicit detection with confidence low. That's the correct behavior — short inputs with faint reasoning hints land at the bottom of the scale, not absent.
