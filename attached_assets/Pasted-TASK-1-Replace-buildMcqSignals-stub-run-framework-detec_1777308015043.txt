1.4 Graceful degradation — if extractSignals fails on an MCQ turn
The free-response branch wraps extractSignals in nothing — failures propagate up. For MCQ, falling back to the old buildMcqSignals stub on signal extractor failure is better than producing no signals at all (the old behavior was at least non-crashing).
Wrap the new MCQ extractSignals call in a try/catch:
tslet evidenceLog: DecisionEvidenceLog;
try {
  const signalStart = Date.now();
  evidenceLog = await extractSignals(context);
  // ... post-process tradeoff floor here ...
  // ... turn event logging here ...
} catch (err) {
  console.error("[director] MCQ signal extraction failed; falling back to mcq_signature stub:", err);
  const fallbackSignals = buildMcqSignals(context.studentInput, decisionPoint);
  evidenceLog = {
    signals_detected: fallbackSignals,
    rds_score: null,
    rds_band: null,
    competency_evidence: mapCompetencyEvidence(fallbackSignals),
    raw_signal_scores: { /* ... */ },
    isMcq: true,
  };
  // log fallback turn event with source: "mcq_signature_fallback"
}
Apply the same pattern to free-response if you want symmetry, but it's not required — free-response failure has always propagated up.
